# Slide 07 — Brainstorming, design before code

**Layout:** Content+Code
**Background:** paper
**Target duration:** 75 seconds

## Eyebrow (optional, ≤24 chars, all-caps)

SUPERPOWERS · SKILL 1

## Headline (sentence case, ≤80 chars)

Turn an idea into a written spec before a single file changes

## Body (3–5 bullets max, OR a key code/command block, OR a comparison table)

- Hard gate: no code, no scaffolding until a design is presented and approved
- One question at a time; explores purpose, constraints, success criteria
- Always proposes 2–3 approaches with trade-offs and a recommendation
- Final artefact is a spec at `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- Terminal next step is `writing-plans` — the skill hands off, never branches

```text
brainstorming → spec.md → writing-plans → plan.md → execution
```

## Optional: one short quote from the source (≤30 words)

> "Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it." — brainstorming SKILL.md

## Speaker notes (2–3 sentences)

The instinct after typing a prompt is to watch code appear; brainstorming kills that instinct and saves an hour every time. Stress the hard gate — even on "simple" tasks the design step costs minutes and prevents the rebuild after lunch. Tee up the next slide as the natural sequel: spec becomes plan.
