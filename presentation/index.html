Turn the thinking budget up when the problem deserves it.

Persistent context

CLAUDE.md — instructions that travel with the repo.

  • Markdown at the project root, loaded into every session before the first prompt
  • Four scopes load top-down — managed, user, project, local; subdirectory files add on for the areas you touch
  • Keep them lean and layered — bloated context degrades performance
  • Review every few months — prune constraints the model has outgrown
  • Run /init to scaffold one, then be specific: "Use 2-space indentation" beats "Format code properly"

Auto memory

Claude takes notes between sessions.

  • A per-repo memory directory with a MEMORY.md index
  • MEMORY.md's first 200 lines load every session; topics load on demand
  • Claude keeps what matters — build commands, gotchas, your stated preferences
  • Toggle with /memory, or set autoMemoryEnabled: false

Superpowers · skill 2

Writing-plans — a step-by-step rhythm.

  • Built for an engineer with "zero context" — every step is explicit
  • File map first: every file to create or modify, paths and responsibilities
  • Steps are 2–5 minutes each: test, run red, code, run green, commit
  • No "TBD" — every code block, command, and expected output is filled in

Superpowers · skill 3

Test-driven development as a non-negotiable.

  • Iron law — no production code without a failing test first
  • Verify red — a test that passes immediately proves nothing
  • Green means minimal code; refactor only after green is observed twice
  • Bug fix? Write the reproducing test before touching the fix

Superpowers · skill 4

Verification before completion — every time.

  • No completion claims without fresh evidence in the current message
  • Identify the command that proves the claim, run it, read output, then claim
  • Linter passing is not the build passing; "done" is not the diff being correct
  • Red flags: "should work", "probably", "looks correct", "great, perfect"

Browser automation

Chrome DevTools MCP or Playwright MCP?

Two ways to drive a browser for automation — pick by reliability, cost, and how much a human stays in the loop.

For inspecting

Chrome DevTools MCP

  • Drives & inspects Chrome over the DevTools Protocol
  • Lighthouse audits and performance traces
  • Reads console, network, and DOM state
  • Lower token cost; no extension needed
For scripting

Playwright MCP

  • Headless, scripted, deterministic — runs in CI
  • Best for long workflows and test suites
  • Cross-browser: Chrome, Firefox, WebKit
  • Engineered storage state for auth

Parallel work

Git worktrees — one repo, many sessions.

  • A worktree is a separate working directory sharing one repo history
  • File-level isolation: edits in one session never touch files in another
  • Each worktree is its own branch + directory
  • .worktreeinclude auto-copies local files like .env into new worktrees
