Opus 4.8 controls

Turn the thinking budget up when the problem deserves it.

  • Five levels — low · medium · high · xhigh · max; high is the Opus 4.8 default everywhere, Claude Code included
  • Start at xhigh for coding & agentic work; drop to medium/low only when evals show quality holds
  • Reach for max on the hardest problems — a concurrency bug, a refactor, an EF Core migration
  • ultracode is a Claude Code menu mode, not an API level — it pins xhigh and triggers a dynamic workflow when warranted

Persistent context

CLAUDE.md — instructions that travel with the repo.

  • /init writes the first CLAUDE.md by reading your repo — you don't hand-author it
  • It captures only the non-obvious: build gotchas, repo etiquette, “money is integer cents” — not “write clean code”
  • Claude keeps it current — auto memory, a Stop hook that proposes edits, /memory to review; you prune, not transcribe
  • Deep, area-specific knowledge lives in skills and path-scoped rules, loaded on demand
  • Emphasis still sticks — IMPORTANT, YOU MUST, NEVER are followed more reliably

Auto memory

Claude takes notes between sessions.

  • A per-repo memory directory with a MEMORY.md index
  • MEMORY.md's first 200 lines load every session; topics load on demand
  • Claude keeps what matters — build commands, gotchas, your stated preferences
  • Say “remember to…” and Claude saves the note; review or edit entries — and toggle the feature — with /memory

Slash commands

A handful of commands you'll reach for daily.

  • /ide — pair the CLI with VS Code or a JetBrains IDE like Rider or IntelliJ: diffs open in the editor, and its diagnostics + current selection flow back to Claude
  • /goal — set a verifiable finish line; Claude works across turns until a checker confirms it's met

Superpowers · skill 2

Writing-plans — a step-by-step rhythm.

  • Built for an engineer with "zero context" — every step is explicit
  • File map first: every file to create or modify, paths and responsibilities
  • Steps are 2–5 minutes each: test, run red, code, run green, commit
  • No "TBD" — every code block, command, and expected output is filled in

Superpowers · skill 3

Test-driven development as a non-negotiable.

  • Iron law — no production code without a failing test first
  • Verify red — a test that passes immediately proves nothing
  • Same loop, any layer: xUnit/NUnit units, bUnit for Blazor components, Testcontainers + Aspire for integration & E2E
  • Bug fix? Write the reproducing test before touching the fix

Superpowers · skill 4

Verification before completion — every time.

  • No completion claims without fresh evidence in the current message
  • Run the proving command — dotnet test, dotnet build, a bUnit render — read the output, then claim
  • Green analyzers aren't a green build; “done” isn't the diff being correct
  • /goal "dotnet test exits 0" makes the finish line machine-checkable — a checker model verifies before Claude stops

Author your own

Write your own skill — a folder and a SKILL.md.

  • A skill is a folder with a SKILL.md — YAML frontmatter plus Markdown steps — in .claude/skills/ (repo) or ~/.claude (everywhere)
  • Reach for one when you keep re-pasting a checklist, or a CLAUDE.md section has become a procedure rather than a fact
  • First, don't reinvent it — browse the marketplaces with /plugin (Anthropic's official catalog plus community ones) and list what already ships
  • Skip the boilerplate — Anthropic's verified /skill-creator scaffolds the folder and SKILL.md, then helps you eval, improve, and benchmark it
  • The description is the trigger — say what it does and when to use it; Claude auto-loads it, or you run /skill-name
  • Keep it lean — under ~500 lines — and push detail into reference files that load on demand

Effectiveness of HTML

Ask for HTML, not Markdown — and stay in the loop.

  • HTML carries what Markdown can't — tables, SVG diagrams, spatial layout, and live interactions
  • Render plans, PR reviews with color-coded diffs, dashboards, and mockups — then open them in the browser
  • Two-way by default: sliders and knobs to explore, copy as JSON or diff to bridge back to text
  • How to start: just say “make an HTML file”, view it, then ask for comparisons or export buttons

LSP · .NET

Semantic code navigation, not grep.

  • Claude Code speaks LSP — the same protocol your IDE uses; add a server per language and it navigates by meaning, not text
  • The official csharp-lsp plugin wraps a Roslyn server: go-to-definition, find-references, hover types, live diagnostics
  • Finds every caller in milliseconds instead of grepping the whole solution — fewer tokens, fewer wrong guesses on refactors
  • Anthropic-verified plugin — add the Roslyn-based csharp-ls server once, then every session navigates by symbol

Parallel work

Git worktrees — one repo, many sessions.

  • A worktree is a separate working directory + branch sharing one repo history — edits in one session never touch another
  • .worktreeinclude auto-copies local files like .env into new worktrees
  • Monorepo: scope context with per-directory CLAUDE.md — root rules plus one file per package — and start Claude in the package you're touching
  • worktree.sparsePaths checks out only the directories a task needs; symlinkDirectories shares one node_modules

Compose them

Skills set the rules. Workflows run them at scale.

  • Skills supply the gates — brainstorm before code, TDD, verification before completion
  • Workflows supply the engine — a script orchestrates subagents, holding the loop + results
  • The working pattern — a workflow whose phases each invoke a superpowers:* skill, so a 500-file migration keeps the discipline
  • Same rule with team agents — no native superpowers support yet (issue #429), so the lead runs every skill, teammates implement, reviewers verify
